ASP.NET Core Swagger Documentation

Eduard Stefanescu
25-08-2021
documentation, dotnet, dotnetcore, swagger

Documentation is one of the greatest way to transfer knowledge about an entire project or a particular part of it. The documentation can help somebody else to understand how it works and can be used also like a reference, when somebody wants to check if his or her assumptions are correct.
But the main advantage of creating the documentation before or during the development time is that can bring to the surface some problems or ideas which no one would have thought of.

So this article is about web API documentation using the Swagger framework, the first part contains a brief introduction and in the second part the actual implementation of it will be explained. There will be another article about Swagger documentation with Bearer Token authentication.

Introduction

Swagger is a part of the OpenAPI Initiative it can be used by other languages too, like Java Spring, Python and many others. And it has multiple tools, one of them is Swagger Hub that can be used to describe Web APIs without using a programming language to describe it; or Swagger Inspector which is a tool to test the API endpoints. But this article is focusing on Swagger Specification using ASP.NET.
The Swagger Specification framework is using a generated documentation by the .NET, which I will explained later how to enabled it, and a UI component that actually illustrates the Web API endpoints, which includes:

  • HTTP request method (i.e. GET, POST);
  • parameters and query parameters;
  • response codes (i.e. OK, NotFound);
  • a description about the endpoint, parameters and return type;

Prerequisites

For this article the default ASP.NET Core project was generated, that means the WeatherForecastController will be used in order to describe the API.
As was previously mentioned a file is needed by the Swagger Framework in order to create the documentation. So to generate this file, the GenerateDocumentationFile property in the ASP.NET Core project must be set to true.

By enabling this in the project settings file (i.e. csproj), then when the project is being built, the compiler will search for the XML comments and will generate an XML file containing all the written comments <a href="#reference1">[1]</a>, in the following format.

We don’t have to bother too much how this XML file will look like, it’s just for explanation purposes.
In the top part, the assembly name is found, meaning that if there are multiple projects, then there will be an XML file for each of it, but only if the GenerateDocumentationFile is enabled. And then all the members that has XML comments are listed, in this case only the constructor of the WeatherForecastController class is described, containing a brief summary. But if the description would have been long, containing multiple keys, like param or response then these would also have been included.
After enabling this property and rebuilding the project, a few warnings will be shown, telling that some type or members has to been described, as is shown in the picture below. These warnings are also from classes that doesn’t need to be included in the API documentation, like the Startup or the Program classes, so in order to disable the warning from these classes, the pragma preprocessor directive must be used, as described below.

The pragma directive takes a few parameters, so there is the pragma-name in this case the warnings that have to be disabled, and then the arguments: disable and 1591 [2]. The last argument is the warning number, which can be found in the Error List window.
Once that the useless warnings were disabled, only the useful ones will be seen. In this from the WeatherForecastController class.

After generating the XML file and removing unnecessary warnings, then two packages needs to be installed in order to generate the Swagger documentation and to visualize it in a more readable way, using an Web page.

  • Swashbuckle.AspNetCore.Swagger
  • Swashbuckle.AspNetCore.SwaggerUI

Writing XML comments

The XML comments needs to clear and to provide enough information in order for those who want to use the API to understand it without having doubts. There might be some misconceptions about writing comments, but in this case writing these comments will help others to understand what the endpoint does and how to use it, because in most cases they will not have access to the code.
Here are some popular websites that published their API Documentation:

  • https://api.stackexchange.com/docs
  • https://www.reddit.com/dev/api/
  • https://developer.twitter.com/en/docs/api-reference-index

The result

After doing all the above steps, now the documentation web page can be used by accessing http://localhost:5000/swagger/index.html. In the picture below is landing page with the WeatherForecast panel opened.

There is the Try it out button through which the endpoint can be tested, this is like testing the endpoint with Postman, Insomnia or any other REST Client, but much more easily, without having to introduce anything. We’ll see that in the next article where the Bearer Authentication will be added that we’ll have to authenticate only one time.
